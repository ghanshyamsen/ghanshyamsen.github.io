SOLID Principles in Practice: Writing Maintainable Code

Practical guidelines that prevent your codebase from becoming unmaintainable.

SOLID principles aren't academic theoryâ€”they're practical guidelines that prevent your codebase from becoming unmaintainable.

S - Single Responsibility Principle

"A class should have only one reason to change."

// Good - Each class has one responsibility
class User {
    public string $name;
    public string $email;
}

class UserRepository {
    public function save(User $user): void { /* database */ }
}

class UserNotificationService {
    public function sendWelcomeEmail(User $user): void { /* email */ }
}

O - Open/Closed Principle

"Open for extension, closed for modification."

// Good - Extend without modifying
interface PaymentGateway {
    public function charge(float $amount): PaymentResult;
}

class StripeGateway implements PaymentGateway {
    public function charge(float $amount): PaymentResult { /* Stripe */ }
}

class PayPalGateway implements PaymentGateway {
    public function charge(float $amount): PaymentResult { /* PayPal */ }
}

L - Liskov Substitution Principle

"Subclasses should be replaceable for their base classes."

I - Interface Segregation Principle

"Clients shouldn't depend on interfaces they don't use."

// Good - Segregated interfaces
interface Workable {
    public function work(): void;
}

interface Eatable {
    public function eat(): void;
}

class Human implements Workable, Eatable { /* both */ }
class Robot implements Workable { /* only work */ }

D - Dependency Inversion Principle

"Depend on abstractions, not concretions."

// Good - Depend on abstraction
class OrderService {
    public function __construct(
        private DatabaseInterface $database
    ) {}
}

// Now easily swappable
$service = new OrderService(new MySQLDatabase());
$service = new OrderService(new MongoDatabase());

Quick Reference

  • S - One class, one job
  • O - Extend, don't modify
  • L - Subclasses are substitutable
  • I - Small, focused interfaces
  • D - Depend on abstractions

Conclusion

SOLID principles lead to code that's easier to test, maintain, and extend. Apply them pragmatically based on your project's complexity.
